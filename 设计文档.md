## 2. 逻辑设计（关系模式）

### 2.1 关系模式转换

依据E-R图转关系模式规则：

**Person(人员)**
```sql
Person(pid, name, gender, birthday, phone, email, address, id_card)
```
- 主键：pid

**Organization(组织)**
```sql
Organization(oid, name, type, address, description)
```
- 主键：oid
- 唯一约束：name

**SocialRelation(社会关系)** — m:n自联系独立模式
```sql
SocialRelation(rid, person1_id, person2_id, relation_type, start_date, description)
```
- 主键：rid
- 外键：person1_id → Person(pid)，person2_id → Person(pid)
- 非空约束：person1_id, person2_id, relation_type

**SocialDuty(社会职责)** — m:n交叉联系独立模式
```sql
SocialDuty(did, person_id, org_id, position, start_date, end_date, description)
```
- 主键：did
- 外键：person_id → Person(pid)，org_id → Organization(oid)
- 非空约束：person_id, org_id, position

### 2.2 函数依赖分析

**Person**
```
pid → name, gender, birthday, phone, email, address, id_card
候选键：{pid}，{id_card}
所有非主属性完全函数依赖于候选键 → 满足2NF
无非主属性传递依赖 → 满足3NF
```

**Organization**
```
oid → name, type, address, description
候选键：{oid}，{name}
满足3NF
```

**SocialRelation**
```
rid → person1_id, person2_id, relation_type, start_date, description
候选键：{rid}
(person1_id, person2_id, relation_type) 可作候选键
满足3NF
```

**SocialDuty**
```
did → person_id, org_id, position, start_date, end_date, description
候选键：{did}，(person_id, org_id, position)
满足3NF
```

### 2.3 规范化验证

所有关系模式均满足3NF要求，设计过程中未发现BCNF违规需要进一步拆分的情况，因此保留在3NF级别，兼顾了规范化程度和查询性能。

### 2.4 视图设计

**人员详细信息视图**
```sql
CREATE VIEW PersonDetail AS
SELECT p.*, 
       GROUP_CONCAT(DISTINCT o.name) AS organizations,
       GROUP_CONCAT(DISTINCT sd.position) AS positions
FROM Person p
LEFT JOIN SocialDuty sd ON p.pid = sd.person_id
LEFT JOIN Organization o ON sd.org_id = o.oid
GROUP BY p.pid;
```

## 3. 物理设计

### 3.1 数据库选型
- DBMS：SQLite 3
- 存储方式：单文件数据库（social_network.db）
- 原因：无需安装数据库服务器，零配置，适合课程设计场景

### 3.2 索引设计

| 表名 | 索引列 | 类型 | 说明 |
|------|--------|------|------|
| Person | name | B-tree | 加速按姓名搜索 |
| Person | id_card | 唯一索引 | 保证身份证号唯一 |
| Organization | name | 唯一索引 | 保证组织名称唯一 |
| SocialRelation | (person1_id, person2_id) | B-tree | 加速关系查询 |
| SocialRelation | relation_type | B-tree | 加速按类型筛选 |
| SocialDuty | person_id | B-tree | 加速按人员查询职责 |
| SocialDuty | org_id | B-tree | 加速按组织查询人员 |

### 3.3 存储策略
- 外键启用级联删除（CASCADE），当删除人员时自动清除其关联关系和职责
- 字符编码：UTF-8
- 密码字段使用 werkzeug 哈希存储

### 3.5 关系穿透分析算法设计

核心算法：基于广度优先搜索（BFS）的路径查找

```
输入：person_a, person_b, max_depth
输出：所有长度≤max_depth的关系路径

算法步骤：
1. 以person_a为起点，构建邻接表（从SocialRelation表中提取）
2. 使用BFS逐层扩展，记录访问路径
3. 每层检查是否到达person_b
4. 到达则记录路径，继续搜索（直到max_depth层）
5. 返回所有找到的路径
```

### 3.6 权限控制设计

采用装饰器模式实现权限验证：

```python
def permission_required(level):
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            user_level = session.get('permission', 0)
            if user_level < level:
                return jsonify({'error': '权限不足'}), 403
            return f(*args, **kwargs)
        return decorated_function
    return decorator
```
